# Hardware Guide

The terminal is hand-built around a Raspberry Pi Pico W on a breadboard: a
4" SPI TFT for the display, an I2S MEMS microphone for capture, two push
buttons, an RGB status LED, and a passive buzzer. Everything runs off USB
power; no soldered carrier board or custom PCB is required.

![Wiring diagram](images/wiring-diagram.svg)

## Wiring

All pin assignments live in
[`firmware/pico_voice_agent/config.h`](../firmware/pico_voice_agent/config.h);
this table is the same information with the reasoning attached.

### Display — ST7796 over SPI0

Write-only SPI at 40 MHz (drop to 27 MHz if you see glitches). MISO is not
connected; the panel is never read back.

| TFT pin | Pico pin | Signal |
|---------|----------|--------|
| SCK     | GP2      | SPI0 clock |
| SDI/MOSI| GP3      | SPI0 data out |
| CS      | GP5      | chip select |
| DC/RS   | GP6      | data/command select |
| RESET   | GP7      | hardware reset |
| VCC     | 3V3      | power |
| GND     | GND      | ground |
| LED     | 3V3      | backlight (always on) |

Note: TFT_eSPI is configured at **compile time inside the library**, not in
the sketch — see [`firmware/README.md`](../firmware/README.md).

### Microphone — SPH0645 over I2S (custom PIO receiver)

The mic occupies five consecutive GPIOs, **including its power pin**: 3V is
supplied from GP22 driven high, and channel-select is GP18 driven low
(left/mono). The SPH0645 draws well under a GPIO's current budget, so
powering it from a pin keeps the wiring to a single 5-pin row.

| Mic pin | Pico pin | Signal | Direction |
|---------|----------|--------|-----------|
| SEL     | GP18     | channel select, held LOW = left/mono | out |
| LRCL    | GP19     | word select (LRCLK), generated by PIO | out |
| DOUT    | GP20     | serial data, sampled by PIO | in |
| BCLK    | GP21     | bit clock, generated by PIO | out |
| 3V      | GP22     | power, held HIGH | out |
| GND     | GND      | ground | — |

**Why a custom PIO receiver:** this pin order puts LRCLK *before* BCLK with
DOUT between them. The stock arduino-pico I2S library requires
`LRCLK == BCLK + 1`, which this layout can't satisfy. The RP2040's PIO
doesn't care about adjacency — the hand-assembled program in
[`pico_i2s.pio.h`](../firmware/pico_voice_agent/pico_i2s.pio.h) drives BCLK
from side-set, LRCLK from SET, and samples DOUT from IN, each on an
independent pin. A DMA channel streams the captured words into a static ring
buffer with zero CPU involvement.

**SPH0645 quirks handled in software** (tunable in
[`pico_i2s.cpp`](../firmware/pico_voice_agent/pico_i2s.cpp)):
- 1-bit clock delay → `SPH0645_DELAY_SHIFT`
- 18 valid bits, MSB-first → `I2S_VALID_BITS`
- quiet output → `I2S_GAIN_SHIFT`

### Controls, LED, buzzer

| Component | Pico pin | Notes |
|-----------|----------|-------|
| Button 1 (push-to-talk) | GP15 | to GND, active-low, internal pull-up, 40 ms debounce |
| Button 2 (reserved) | GP14 | same wiring; unused, free for a future feature |
| WS2812 RGB LED | GP12 | single pixel, data in; VCC to 3V3, GND to GND |
| Passive buzzer | GP13 | driven with `tone()`; passive type is required (an active buzzer ignores frequency) |

### Status LED legend

| State | LED | Sound |
|-------|-----|-------|
| Idle / ready | solid dim green | — |
| Recording | solid red | short high beep on start |
| Processing | blinking blue | short beep on stop |
| Response | green flash | low chime |
| Error | blinking amber | long low buzz |

## Power

Everything runs from USB 5 V through the Pico's onboard 3V3 regulator. Worst
case draw (Wi-Fi TX + backlight + LED) stays comfortably inside a standard
USB port's budget. There is no battery; the device is meant to sit on a desk.

## Memory design notes

The RP2040 has 264 KB of SRAM and no PSRAM, which shaped the firmware:

- **No full-screen framebuffer.** 480x320 at 16 bpp would be ~300 KB — more
  than all of SRAM. The UI draws straight to the panel.
- **One static PCM buffer + one static DMA ring.** The audio path never
  allocates per-chunk; capture-to-network is a fixed 2 KB ring plus a 512 B
  chunk buffer.
- **TLS is the tightest moment.** The WSS handshake needs tens of KB
  transiently; the socket is opened once and kept alive across turns.
- **No `String`** in the audio/network path — fixed `char[]` + `snprintf`.
